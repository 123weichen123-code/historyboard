# 前端设计文档（H5）

## 1. 技术栈

- 原生 HTML/CSS/JavaScript
- 面向移动优先，兼容桌面端
- 通过 REST API 与 Java 服务通信

## 2. 信息架构

1. 顶部 Hero 区
   - 产品标题
   - 一句话说明
   - 搜索框与操作按钮
   - 快速建议词
2. 主体 Dashboard
   - 左：搜索结果列表
   - 右：事件详情 + 同期世界事件
3. 下方 Time Strip
   - 中国历史时间带快速入口

## 3. 页面交互流程

1. 首次进入：调用 `/api/history/discover` + `/api/history/timeline`
2. 用户输入关键词：调用 `/api/history/search`
3. 选择某条结果：调用 `/api/history/events/{event_id}`
4. 点击时间带节点：复用搜索流程

## 4. 组件设计

- SearchBar
  - 输入、检索按钮、随机按钮
- SuggestionChips
  - 热门词与推荐词
- ResultList
  - 可点击事件卡片，支持选中高亮
- DetailCard
  - 事件核心信息
- WorldContextList
  - 同期世界事件卡片列表
- TimelineStrip
  - 时序入口按钮网格

## 5. 状态管理（轻量）

- `state.query`
- `state.results`
- `state.selectedId`
- `state.quickQueries`
- `state.timeline`

## 6. 可用性与视觉策略

- 强可读排版：标题与正文层级明显
- 搜索反馈即时：无结果状态、错误状态明确
- 时间与标签结构化展示，降低理解成本
- 保留扩展位：关系图、来源卡片、筛选器

## 7. 时间轴可视化方案（已落地第一版）

### 7.1 目标

把原先的“时间带入口按钮”升级为真正可浏览的连续时间轴，让用户可以在一条轴线上快速理解：

- 关键事件的先后顺序
- 事件所处朝代/类别
- 当前选中事件在整体历史中的位置

### 7.2 交互方案

1. 默认展示一条横向连续时间轴
   - 以年份区间为基准铺开节点
   - 节点上下交错排布，避免文本互相遮挡
2. 每个节点包含三层信息
   - 事件标题
   - 年代/时期
   - 朝代或类别标签
3. 点击节点
   - 直接触发检索并切换详情区
   - 同步刷新右侧事件详情与“同期世界关键节点”
4. 选中态高亮
   - 当前详情事件在时间轴上高亮显示
   - 自动滚动到可视区域中央附近
5. 密度切换
   - 提供“紧凑 / 标准 / 展开”三档，用于控制时间轴横向展开宽度
6. 数据兜底
   - 若缺少年份字段，则退回为卡片网格模式，保证页面可用

### 7.3 视觉策略

- 主轴线使用连续横线，表达历史长河感
- 年份刻度做轻量标记，避免喧宾夺主
- 节点卡片采用上下错位布局，形成“编年史”阅读节奏
- 当前选中节点用强调色和外圈高亮，增强定位感

### 7.4 第一版边界

- 当前支持三种视图：中国单轴 / 世界单轴 / 中外并行同轴对照
- 世界事件可单独点击查看详情
- 当前并行模式为“同一条轴线上的双来源节点”，还不是上下双轨双时间轴
- 后续可继续加入筛选、缩放、关系线、专题模式

### 7.5 工作量评估（已验证并完成）

1. 数据层
   - 需要把时间轴接口从“仅中国”扩展为“支持中国 / 世界 / 全部”
   - 需要让世界事件也能单独查看详情
2. 前端层
   - 需要把单一 time strip 改成带范围切换、筛选器、密度控制的时间轴组件
   - 需要给中国节点和世界节点建立不同视觉编码
3. 联调风险
   - 主要风险不在算法，而在事件类型差异（朝代 vs 区域）与详情展示兼容
   - 属于中等工作量、低到中等风险，适合直接一次性落地 MVP

## 8. 扩展规划

- 增加筛选面板（朝代、类别、年份范围）
- 增加“双时间轴 / 中外并行时间轴”模式
- 增加事件关系图与来源引用区
